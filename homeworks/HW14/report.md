# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

В работе использовалась небольшая база знаний из текстовых документов по теме NLP и RAG.

Документы были разбиты на чанки фиксированного размера с перекрытием.  
Для получения векторных представлений использовалась модель `all-MiniLM-L6-v2`.

На основе эмбеддингов был построен индекс FAISS и реализован retrieval.  
Проведена оценка качества (hit@k, recall@k).

База знаний была расширена новыми документами, после чего выполнена переиндексация и повторный анализ retrieval.

Также реализован простой mini-RAG, который формирует ответ на основе найденных фрагментов.

## 3. База знаний и подготовка документов

- Предметная область: NLP, retrieval, RAG  
- Источник документов: вручную созданные текстовые файлы  
- Число исходных документов: 10  
- Число чанков после разбиения: 10  
- Способ чанкинга: разбиение текста на фрагменты фиксированной длины  
- Параметры: chunk_size=400, overlap=100  

Комментарий:  
База знаний содержит определения и объяснения ключевых понятий, что делает её удобной для задач retrieval.  
Документы небольшие, поэтому каждый из них даёт один чанк.  
Тематика однородная, что упрощает поиск релевантных фрагментов.  
Такая база подходит для демонстрации mini-RAG, так как ответы можно формировать напрямую из текста.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- Модель: sentence-transformers/all-MiniLM-L6-v2  
- Метрика сходства: косинусное сходство (через нормализацию + Inner Product)  
- Индекс: FAISS IndexFlatIP  
- top_k: 3  

Первые примеры показали, что система корректно находит релевантные документы для простых запросов.

---

### 4.2. Контрольные запросы и оценка retrieval

- Количество запросов: ~8  
- Релевантный ответ: документ, содержащий нужное определение  
- Метрики: hit@k, recall@k  

- Итоговый hit@k: ~1.0  
- Итоговый recall@k: ~1.0  

Самые простые запросы:
- "What is FAISS?"
- "What is Python?"

Самые проблемные:
- вопросы с близкими по смыслу темами (retrieval vs RAG)

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

Сравнивался параметр: top_k  
Варианты: top_k=1 и top_k=3  

Результат:
- при увеличении top_k повышается вероятность найти нужный документ  
- но увеличивается шум  

Выбранный вариант: top_k=3 — баланс между точностью и полнотой.

Обновление базы:
- добавлены документы про metadata и hallucinations  

Переиндексация:
- пересчитаны эмбеддинги  
- пересоздан индекс FAISS  

После обновления:
- retrieval стал находить новые темы  
- изменились результаты для новых запросов

## 7. Анализ

Выбранная база знаний оказалась удачной для учебной задачи, так как содержит чёткие определения.  
Чанкинг почти не повлиял на результат, так как документы короткие.  
Формальная оценка retrieval показала высокие значения метрик, что подтверждает корректность пайплайна.  

Эксперимент с top_k показал, что увеличение числа возвращаемых документов улучшает полноту, но может ухудшить точность.  

После обновления базы знаний retrieval начал корректно обрабатывать новые темы, что подтверждает важность переиндексации.  

Mini-RAG показал зависимость от качества retrieval: если найден правильный документ, ответ корректный.  

Основные ошибки связаны с:
- похожими по смыслу документами  
- простым способом генерации ответа  

Ограничения:
- маленькая база знаний  
- отсутствие LLM  
- примитивная генерация ответа

## 8. Итоговый вывод

В качестве базового решения подходит pipeline:  
чанкинг → эмбеддинги → FAISS → top_k retrieval → сбор контекста.

Главный вывод: качество retrieval напрямую определяет качество ответа в RAG.

Также стало понятно, что даже простой mini-RAG может работать корректно при хорошей базе знаний и настройке retrieval.
